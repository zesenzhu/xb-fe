# Git 规范化提交规约 (commit-convention)

本技能包定义了统一的 Git 提交信息规范。当您在本项目中完成代码编写并准备执行 `git commit` 时，**必须严格按照本规约进行提交信息的撰写**。所有提交信息均需使用中文表述，做到条理清晰、改动可追溯。

---

## 1. 提交信息结构格式

所有的 Commit Message 必须遵循以下三段式结构：

```
<type>(<scope>): <简洁的中文标题>

- [修改点1] 具体到文件名/模块名：详细描述做了什么修改以及为什么修改
- [修改点2] 具体到文件名/模块名：详细描述做了什么修改以及为什么修改
- [修改点3] ...
```

> **注意**：
> 1. 首行（Header）与正文（Body）之间必须保留一个**空行**。
> 2. 标题字数建议控制在 50 字以内，结尾不加句号。
> 3. 正文部分必须详细列出所有涉及的修改点。

---

## 2. 核心元素定义

### 2.1 提交类型 (Type)
首行的 `type` 必须使用以下常规的英文标识，用于清晰划分改动性质：
*   `feat`: 引入新功能（New Feature）
*   `fix`: 修复 Bug 或构建报错（Bug Fix）
*   `docs`: 仅文档、README 的修改（Documentation）
*   `style`: 代码格式化、缺失分号补齐等（不影响核心业务逻辑的变动）
*   `refactor`: 代码重构（不增加新功能，也不修复 Bug 的代码变动）
*   `perf`: 提升性能、减少内存占用或优化加载速度（Performance）
*   `test`: 增加、修改或修复单元测试、集成测试（Test）
*   `chore`: 构建工具、CI/CD 流程、依赖库更新、辅助脚本修改（Chore）

### 2.2 影响范围 (Scope) - 可选
用括号包裹，标明此次提交主要影响的子模块或业务组件。例如：`(user-page)`、`(auth)`、`(middleware)` 等。

### 2.3 标题 (Subject)
使用简短的中文，说明此次提交最核心的变化。

### 2.4 正文修改点 (Body)
**这是本规约最核心的要求。** 正文部分绝对不能空缺，且不能只写一句笼统的话。必须按条目详细列出所有受影响的文件以及具体的逻辑改动。

*   **格式要求**：每行以 `-` 开头，指明文件路径或模块名，接着是改动细节。
*   **内容要求**：
    *   描述**“改了哪里”**、**“怎么改的”**、**“解决了什么问题”**。
    *   如果有语法错误修复，必须列出原报错位置和修复方式。
    *   如果有新增的弹窗或状态，需写明弹窗的名称和触发逻辑。

## 3. 正确与错误示范

### 3.1 正确示范 🟢

```
feat(user-page): 增加用户管理编辑与改密功能

- 前端 `src/app/(admin)/admin/user/page.tsx`：
  1. 新增 EditModal，实现编辑用户账号属性（昵称、邮箱、所属角色及账号状态）并完成表单数据双向回显。
  2. 新增 PasswordModal，实现管理员重置用户密码功能。
  3. 表单内引入“二次输入一致性”以及“最小 6 位长度”的强校验逻辑，未通过校验时拦截提交。
  4. 修复了由于 `handleDeleteUser` 函数外部遗漏闭合大括号 `};` 导致的编译报错。
- 前端 `src/hooks/useTableQuery.ts`：
  - 微调了表格查询的 refetch 触发逻辑，保证在用户属性更新或改密成功后自动刷新表格列表。
```

### 3.2 错误示范 🔴

*   **错误示范 1（缺少具体修改点）**：
    ```
    feat: 新增编辑和改密弹窗
    ```
    *解释*：只有 Title，正文没有详细的修改点和涉及的文件列表，不符合要求。

*   **错误示范 2（描述含糊）**：
    ```
    fix(user): 修复报错

    - 修复了 user 页面的一些语法错误。
    - 调整了按钮。
    ```
    *解释*：描述过于宽泛，没有明确指出是哪个文件的什么语法错误（如遗漏了闭合花括号），也没有说明具体是哪个按钮。

---

## 4. 辅助自检清单

在执行 `git commit` 前，请确保符合以下标准：
1. [ ] 提交信息的第一行是否符合 `<type>(<scope>): <subject>` 规范？
2. [ ] 是否隔开了一个空行？
3. [ ] 正文部分是否使用中文列表详细说明了每一个涉及文件的修改点？
4. [ ] 具体的逻辑改动（例如新增了什么校验、修复了什么报错、调整了什么网络请求）是否表述清晰？
